# Student HATEOAS CRUD API

A Spring Boot REST API for managing students with HATEOAS navigation, validation, centralized error handling, DTO mapping with MapStruct, and logical delete through a student status field.

The API exposes a clean student lifecycle:

- create a student as `ACTIVE`
- read all students or filter by status
- read a single student
- update student data and status
- deactivate a student with `DELETE`, setting the status to `INACTIVE`

## Tech Stack

- Java 17
- Spring Boot 3.3.0
- Spring Web
- Spring Data JPA
- Spring HATEOAS
- Bean Validation
- MapStruct
- Lombok
- MySQL
- H2 for tests
- Maven Wrapper

## Domain Model

Each student has:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | `Long` | Generated by the database |
| `name` | `String` | Required |
| `surname` | `String` | Required |
| `age` | `Integer` | Required, between 1 and 120 |
| `email` | `String` | Required, valid email format |
| `status` | `StudentStatus` | `ACTIVE` or `INACTIVE` |

New students are created with `ACTIVE` status by default.

## API Base URL

```text
http://localhost:8080/api/v1/students
```

The plural route is the canonical API path. The singular route `/api/v1/student` is kept as a compatibility alias.

## Endpoints

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/api/v1/students` | List all students |
| `GET` | `/api/v1/students?status=ACTIVE` | List active students |
| `GET` | `/api/v1/students?status=INACTIVE` | List inactive students |
| `GET` | `/api/v1/students/{id}` | Get one student |
| `POST` | `/api/v1/students` | Create a student |
| `PUT` | `/api/v1/students/{id}` | Update a student |
| `DELETE` | `/api/v1/students/{id}` | Deactivate a student |

## HATEOAS

Responses include `_links` so clients can discover valid next actions without hardcoding the entire workflow.

Collection responses use `_embedded.students`:

```json
{
  "_embedded": {
    "students": [
      {
        "id": 1,
        "name": "Marco",
        "surname": "Neri",
        "age": 23,
        "email": "marco.neri@example.com",
        "status": "ACTIVE",
        "_links": {
          "self": {
            "href": "http://localhost:8080/api/v1/students/1"
          },
          "students": {
            "href": "http://localhost:8080/api/v1/students{?status}",
            "templated": true
          },
          "update": {
            "href": "http://localhost:8080/api/v1/students/1"
          },
          "deactivate": {
            "href": "http://localhost:8080/api/v1/students/1"
          }
        }
      }
    ]
  },
  "_links": {
    "self": {
      "href": "http://localhost:8080/api/v1/students{?status}",
      "templated": true
    },
    "all-students": {
      "href": "http://localhost:8080/api/v1/students"
    },
    "filter-by-status": {
      "href": "http://localhost:8080/api/v1/students{?status}",
      "type": "GET",
      "templated": true
    },
    "active-students": {
      "href": "http://localhost:8080/api/v1/students?status=ACTIVE"
    },
    "inactive-students": {
      "href": "http://localhost:8080/api/v1/students?status=INACTIVE"
    }
  }
}
```

Inactive students do not expose the `deactivate` link.

## Request Examples

### Create Student

```bash
curl -X POST http://localhost:8080/api/v1/students \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Marco",
    "surname": "Neri",
    "age": 23,
    "email": "marco.neri@example.com"
  }'
```

### Get All Students

```bash
curl http://localhost:8080/api/v1/students
```

### Filter By Status

```bash
curl "http://localhost:8080/api/v1/students?status=ACTIVE"
```

### Get Student By ID

```bash
curl http://localhost:8080/api/v1/students/1
```

### Update Student

```bash
curl -X PUT http://localhost:8080/api/v1/students/1 \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Marco",
    "surname": "Neri",
    "age": 24,
    "email": "marco.neri@example.com",
    "status": "ACTIVE"
  }'
```

### Deactivate Student

```bash
curl -X DELETE http://localhost:8080/api/v1/students/1
```

## Error Handling

The API returns a common error response for validation errors, malformed JSON, type mismatch, missing resources, data integrity violations, unsupported methods, missing endpoints, and unexpected errors.

Example validation error:

```json
{
  "timestamp": "2026-06-14T16:30:00.000000",
  "status": 400,
  "error": "Bad Request",
  "code": "VALIDATION_ERROR",
  "message": "Request validation failed",
  "path": "/api/v1/students",
  "errors": [
    {
      "code": "NotBlank",
      "field": "name",
      "rejectedValue": "",
      "message": "Name is required"
    }
  ]
}
```

Unexpected errors are logged with stack trace by the global exception handler.

## Local Setup

### Requirements

- Java 17 or newer
- MySQL running locally
- Maven is optional because the project includes Maven Wrapper

### Database

The default local database configuration is:

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/devtalkcodingzone
spring.datasource.username=root
spring.datasource.password=root
```

Create the database before starting the application:

```sql
CREATE DATABASE devtalkcodingzone;
```

If you already have students created before the `status` column was introduced, run:

```sql
UPDATE student SET status = 'ACTIVE' WHERE status IS NULL;
```

## Design Notes

- DTOs are mapped through MapStruct.
- Request models use a shared base class for common student fields.
- `DELETE` is implemented as logical deactivation, not physical deletion.
- HATEOAS links are built through Spring HATEOAS assemblers.
- Errors are normalized through a single `ErrorResponse` contract.
- `spring.jpa.open-in-view=false` is set to keep persistence boundaries explicit.
